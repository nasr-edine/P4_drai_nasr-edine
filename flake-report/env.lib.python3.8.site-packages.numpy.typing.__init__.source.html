3 Typing (:mod:`numpy.typing`)
4 ============================
5  
6 .. warning::
7  
8 Some of the types in this module rely on features only present in
9 the standard library in Python 3.8 and greater. If you want to use
10 these types in earlier versions of Python, you should install the
11 typing-extensions_ package.
12  
13 Large parts of the NumPy API have PEP-484-style type annotations. In
14 addition a number of type aliases are available to users, most prominently
15 the two below:
16  
17 - `ArrayLike`: objects that can be converted to arrays
18 - `DTypeLike`: objects that can be converted to dtypes
19  
20 .. _typing-extensions: https://pypi.org/project/typing-extensions/
21  
22 Differences from the runtime NumPy API
23 --------------------------------------
24  
25 NumPy is very flexible. Trying to describe the full range of
26 possibilities statically would result in types that are not very
27 helpful. For that reason, the typed NumPy API is often stricter than
28 the runtime NumPy API. This section describes some notable
29 differences.
30  
31 ArrayLike
32 ~~~~~~~~~
33  
34 The `ArrayLike` type tries to avoid creating object arrays. For
35 example,
36  
37 .. code-block:: python
38  
39 >>> np.array(x**2 for x in range(10))
40 array(<generator object <genexpr> at ...>, dtype=object)
41  
42 is valid NumPy code which will create a 0-dimensional object
43 array. Type checkers will complain about the above example when using
44 the NumPy types however. If you really intended to do the above, then
45 you can either use a ``# type: ignore`` comment:
46  
47 .. code-block:: python
48  
49 >>> np.array(x**2 for x in range(10)) # type: ignore
50  
51 or explicitly type the array like object as `~typing.Any`:
52  
53 .. code-block:: python
54  
55 >>> from typing import Any
56 >>> array_like: Any = (x**2 for x in range(10))
57 >>> np.array(array_like)
58 array(<generator object <genexpr> at ...>, dtype=object)
59  
60 ndarray
61 ~~~~~~~
62  
63 It's possible to mutate the dtype of an array at runtime. For example,
64 the following code is valid:
65  
66 .. code-block:: python
67  
68 >>> x = np.array([1, 2])
69 >>> x.dtype = np.bool_
70  
71 This sort of mutation is not allowed by the types. Users who want to
72 write statically typed code should insted use the `numpy.ndarray.view`
73 method to create a view of the array with a different dtype.
74  
75 DTypeLike
76 ~~~~~~~~~
77  
78 The `DTypeLike` type tries to avoid creation of dtype objects using
79 dictionary of fields like below:
80  
81 .. code-block:: python
82  
83 >>> x = np.dtype({"field1": (float, 1), "field2": (int, 3)})
84  
85 Although this is valid Numpy code, the type checker will complain about it,
86 since its usage is discouraged.
87 Please see : :ref:`Data type objects <arrays.dtypes>`
88  
89 Number Precision
90 ~~~~~~~~~~~~~~~~
91  
92 The precision of `numpy.number` subclasses is treated as a covariant generic
93 parameter (see :class:`~NBitBase`), simplifying the annoting of proccesses
94 involving precision-based casting.
95  
96 .. code-block:: python
97  
98 >>> from typing import TypeVar
99 >>> import numpy as np
100 >>> import numpy.typing as npt
101  
102 >>> T = TypeVar("T", bound=npt.NBitBase)
103 >>> def func(a: "np.floating[T]", b: "np.floating[T]") -> "np.floating[T]":
104 ... ...
105  
106 Consequently, the likes of `~numpy.float16`, `~numpy.float32` and
107 `~numpy.float64` are still sub-types of `~numpy.floating`, but, contrary to
108 runtime, they're not necessarily considered as sub-classes.
109  
110 Timedelta64
111 ~~~~~~~~~~~
112

113 The `~numpy.timedelta64` class is not considered a subclass of `~numpy.signedinteger`,
114 the former only inheriting from `~numpy.generic` while static type checking.
